info Contexto del Proyecto

¿Por qué modernizar?

La iniciativa no es una simple migración de pantallas, sino una transformación arquitectónica completa.

warning

Problema Actual

Sistema construido en Oracle Forms con alto acoplamiento entre pantallas, lógica y base de datos. Difícil de mantener, escalar y evolucionar.

Legado Oracle Forms Alto acoplamiento
target

Objetivo

Separar responsabilidades, mejorar la experiencia de usuario, reducir la dependencia del legado y habilitar una evolución progresiva del sistema.

Desacoplamiento Evolución progresiva
strategy

Estrategia

Frontend moderno con Angular, design system corporativo @seguros-bolivar/ui-bundle, y backend desacoplado en Java con arquitectura hexagonal.

15 Decisiones Clave

Cada decisión está respaldada por una justificación técnica y un beneficio esperado.

1

Frontend en Angular 21 — MVVM

Angular ofrece un marco sólido para aplicaciones empresariales de gran escala, con política de versiones predecible y soporte LTS por ventanas definidas.

Robustez Mantenibilidad Evolución controlada
2

Design System @seguros-bolivar/ui-bundle

Uso obligatorio del design system corporativo multi-brand como base visual y de componentes reutilizables. Garantiza uniformidad y alineación corporativa.

Consistencia UX Aceleración dev Gobierno visual
3

Arquitectura MVVM en Frontend

Separación clara entre vista, lógica de presentación y modelos de datos. Favorece desacoplamiento, pruebas y mantenibilidad en aplicaciones con alta complejidad funcional.

Menor acoplamiento Escalabilidad
4

Microfrontend por Dominios

Segmentación por capacidades de negocio (no por pantallas). Angular soporta federation para independencia de despliegue y evolución por equipos.

Independencia de despliegue Aislamiento funcional
5

Seguridad — Oracle Access Manager

Autenticación centralizada con SSO, MFA e integración de identidades entre nube y on-premises. No se resuelve localmente en cada módulo.

Sesión unificada Gobierno de perfiles
6

Responsabilidad del Frontend

UX, navegación, composición de vistas, validaciones de entrada, consumo de APIs y estado de pantalla. Sin reglas críticas de negocio.

Reglas en backend Menor riesgo funcional
7

Backend Java 21 — Hexagonal

Aislamiento del dominio de negocio respecto a tecnologías concretas: frameworks, persistencia, REST e integraciones externas.

Testabilidad Mantenibilidad Bajo acoplamiento
8

Estructura Backend: Puertos y Adaptadores

Tres zonas: dominio, aplicación e infraestructura. Puertos y adaptadores para desacoplar exposición, persistencia, seguridad e integración.

Evolución técnica Independencia del dominio
9

Persistencia: JDBC + JPA ORM

Acceso a Oracle mediante JDBC + JPA/Hibernate encapsulado en adaptadores de infraestructura. No ODBC — es el estándar natural para Java empresarial.

Compatibilidad Java Menor complejidad
10

Integración con Legado Oracle Forms

Procedimientos almacenados, PL/SQL, validaciones históricas y estructuras heredadas encapsuladas en adaptadores específicos de infraestructura.

Control de transición Evolución progresiva
11

Modernización, No Réplica

Modernización funcional y arquitectónica. Migrar literalmente solo cambiaría la tecnología sin resolver los problemas estructurales del sistema actual.

Plataforma preparada Resolución estructural
12

Migración Incremental

Migración por capacidades/dominios con convivencia temporal con el legado. Un big bang elevaría el riesgo operativo de un proceso tan crítico como pensiones.

Reducción de riesgo Transición controlada
13

APIs Orientadas a Casos de Uso

El backend expone capacidades por casos de uso del negocio, no como reflejo de tablas o estructuras físicas heredadas de la base de datos.

Alineación con dominio Independencia de persistencia
14

Estrategia de Pruebas

Pruebas unitarias y de integración obligatorias en frontend y backend. Asegurar estabilidad funcional y técnica desde el inicio de la migración.

Migración Incremental

Transición controlada por dominios con convivencia temporal con el legado.

Fase 1 — Fundamentos

Establecer la arquitectura base: proyecto Angular 21 con MVVM, integración del design system @seguros-bolivar/ui-bundle, backend Java 21 con estructura hexagonal y configuración de Oracle Access Manager.

Fase 2 — Primer Dominio Piloto

Migrar el primer dominio funcional completo (ej. Consulta) como piloto. Validar la arquitectura end-to-end, incluyendo integración con legado Oracle mediante adaptadores.

Fase 3 — Expansión por Dominios

Migración progresiva de los dominios restantes: Trámites, Gestión Documental, Liquidación, Administración y Parametrización. Cada dominio como microfrontend independiente.

Fase 4 — Consolidación y Desacople

Reducción progresiva de dependencias con Oracle Forms. Consolidación de APIs por casos de uso, observabilidad completa y estabilización de la plataforma.

Conclusión Ejecutiva

La decisión arquitectónica consiste en modernizar el sistema de pensiones mediante una nueva plataforma empresarial con frontend en Angular 21 bajo MVVM, estandarizada con @seguros-bolivar/ui-bundle, con capacidad de evolucionar a microfrontend por dominios funcionales, e integrada al esquema corporativo de identidad mediante Oracle Access Manager.

En backend, la solución se construirá en Java 21 con arquitectura hexagonal, desacoplando el dominio del negocio de la persistencia y del legado Oracle, y ejecutando una migración incremental que reduzca riesgo y preserve continuidad operativa.
